DESIGN

4.

Explain how design and development intersect. What role do you see a developer playing in the design process?

I think it's very important that designers and developers talk to one another during the design process. I have come up with really nifty designs that were nearly impossible to actually code when I got to the markup stage. My teacher let me find this out the hard way, which was tough at the time but taught me a valuable lesson about planning that I remember each time I work on a design now. Designs should take development feasibility into account, and development should respect the design process and abide by the mockup.

5.

Describe how grids are useful, and how you generally go about using them, or implementing them.

Grids are useful for keeping elements organized and tidy on the page. I start using grids in the design phase. I have a loose grid in place for thumbnails and a measured one for mockups in Photoshop. How many columns and how much gutter I use depends what screen size and what kind of information I'm working with. In the markup phase, I tend to prefer using floats over inline-block to implement grids because I find them a little easier to work with and they hold up on older, less pleasant browsers.

6a.

Talk a little bit about pattern libraries. What are they and how are they useful? Have you used them before?

A pattern library is a system that developers put into place to give a website a unified feel. Everyone working on the site follows the same guidelines so that things like the navigation and buttons look the same across all pages. It also prevents extra work, since designing appropriate buttons once instead of five times saves time and money. This probably also helps in future redesign projects. I have not had the chance to use a pattern library yet, but it sounds efficient so I'm already a fan.

6b.

Talk a little bit about visual systems. What are they and how are they useful? Have you used them before?

A visual system ensures that the site you are designing 'feels' like the same site regardless of whether you are visiting it on your phone, tablet, or computer. Colors, fonts, basic setups, and the like should remain the same. I try to implement a visual system for any website I build. For example, the structure of a navigation bar may be narrower and more stacked on smaller screen sizes, but it remains the same color and the menu options look basically the same regardless of screen size.

10.

What is your comfort level with responsive design? Have you done any mobile-first projects before? Tell us about what you know.

My Iron Yard instructor was adamant about designing for mobile first. I went rogue on one project and coded for tablet first. Predictably, it was a nightmare to fix and I'll never do it again. Teacher: amused. Cat: sheepish.

In that experience, I found that it is much more difficult to fix media queries backwards than to just build them up from smaller screen sizes initially. You can't 'just ignore' mobile either: It is wildly important to create a mobile-friendly site since so much traffic is now on mobile instead of desktop. Google found that in 2015, mobile searches surpassed desktop searches in several countries including the US. It would be terrible to provide a sub-par experience or, possibly worse, to delete content to make it 'fit' onto mobile, for so many users. People expect to have full functionality and full information regardless of what they're using to access your site.

In practice, sometimes I change images resolutions and/or measurement ratios based on screen size. In mobile everything becomes more stacked, and the navigation bar does change so that buttons are easier to click on. It is a nice touch to add a click-to-call links for mobile users as well.

13.

Explain how HTML, CSS, and JS interact - what is each layer responsible for?

HTML provides the backbone and basic information on the site. If you're watching a house being built, it would be the scaffolding. CSS provides the nice visuals and layout of the site. In the HTML house, it would be things like the paint color and walls. JavaScript is all about action. It can do fun things like slide elements over or move them down. In the HTML house, JavaScript might be something like the dimmer switch on a light. It adds an extra layer of cool, but if it breaks you should ideally still have a light.

14.

Progressive enhancement or graceful degradation? Choose your side.

I like the idea of progressive enhancement because, like mobile-first design, it begins from the bottom up. You might as well start simple and build upon that. Just because someone isn't browsing with JavaScript enabled doesn't mean that the internet shouldn't work for them. And trying to build back towards older browsers after writing fancy code seems inefficient.
